# Устранение проблем при деплое

Документ описывает проблемы, которые соответствуют текущему `deploy.sh`, локальному `start_server.py` и фактической структуре проекта.

## Ошибка: AccessDenied при PutBucketAcl

### Причина

Ошибка `AccessDenied` при выполнении `PutBucketAcl` возникает, когда у сервисного аккаунта недостаточно прав для изменения ACL (Access Control List) бакета.

### Решения

#### Решение 1: Назначить роль storage.admin (рекомендуется)

1. Откройте [Yandex Cloud Console](https://console.cloud.yandex.ru/)
2. Перейдите в раздел **"Сервисные аккаунты"**
3. Откройте ваш сервисный аккаунт
4. Перейдите на вкладку **"Роли"**
5. Нажмите **"Назначить роли"**
6. Выберите роль **`storage.admin`** (полный доступ к Object Storage)
7. Нажмите **"Сохранить"**

⚠️ **Важно**: Роль `storage.admin` даёт полный доступ к Object Storage. Если нужен более ограниченный доступ, используйте Решение 2.

#### Решение 2: Настроить публичный доступ через веб-интерфейс (проще)

Публичный доступ можно настроить вручную через веб-интерфейс Yandex Cloud:

1. Откройте [Yandex Cloud Console](https://console.cloud.yandex.ru/)
2. Перейдите в раздел **"Object Storage"**
3. Откройте бакет `practicum-banners`
4. Перейдите в раздел **"Доступ"**
5. Включите следующие опции:
   - ✅ **"Публичный доступ на чтение объектов"**
   - ✅ **"Публичный доступ на чтение списка объектов"**
6. Нажмите **"Сохранить"**

После этого скрипт деплоя будет работать без ошибок, даже если ACL не настроен через CLI.

#### Решение 3: Использовать упрощённый скрипт

В текущем репозитории отдельного `deploy-simple.sh` нет. Если бакет уже настроен и нужна принудительная синхронизация только средствами основного скрипта, используйте:

```bash
./deploy.sh --full
```

Если проблема вызвана локальным кэшем изменений, удалите `.deploy-manifest` и повторите обычный запуск:

```bash
rm -f .deploy-manifest
./deploy.sh
```

### Проверка прав сервисного аккаунта

Проверьте, какие роли назначены вашему сервисному аккаунту:

1. Откройте сервисный аккаунт в Yandex Cloud Console
2. Перейдите на вкладку **"Роли"**
3. Убедитесь, что есть одна из ролей:
   - `storage.admin` — полный доступ (рекомендуется для деплоя)
   - `storage.editor` — редактирование бакетов и объектов
   - `storage.uploader` — только загрузка файлов (недостаточно для настройки ACL)

### Минимально необходимые роли

Для полного деплоя (включая настройку бакета) нужна роль:
- **`storage.admin`** — для настройки ACL, хостинга, CORS

Для только загрузки файлов достаточно:
- **`storage.uploader`** — но тогда настройки нужно делать вручную через веб-интерфейс

## Другие частые ошибки

### Ошибка: "The bucket you are attempting to access must be addressed using the specified endpoint"

**Причина**: Не указан правильный endpoint URL.

**Решение**: Убедитесь, что используете:
```bash
aws --endpoint-url=https://storage.yandexcloud.net s3 ...
```

### Ошибка: "InvalidAccessKeyId"

**Причина**: Неправильный ключ доступа.

**Решение**:
1. Проверьте правильность `aws_access_key_id`
2. Убедитесь, что ключ не был удалён в Yandex Cloud Console
3. Создайте новый ключ, если нужно

### Ошибка: "SignatureDoesNotMatch"

**Причина**: Неправильный секретный ключ.

**Решение**:
1. Проверьте правильность `aws_secret_access_key`
2. Убедитесь, что скопировали ключ полностью, без лишних пробелов
3. Создайте новый ключ, если нужно

### Ошибка: "NoSuchBucket"

**Причина**: Бакет не существует.

**Решение**:
1. Создайте бакет через веб-интерфейс Yandex Cloud Console
2. Или используйте команду:
   ```bash
   aws --endpoint-url=https://storage.yandexcloud.net s3 mb s3://practicum-banners
   ```

### Файлы загружаются, но сайт не открывается

**Причина**: Не настроен хостинг статического сайта или публичный доступ.

**Решение**:
1. Откройте бакет в Yandex Cloud Console
2. Включите **"Публичный доступ на чтение объектов"** в разделе "Доступ"
3. Включите хостинг в разделе "Веб-сайт"
4. Укажите главную страницу: `index.html`

### Изменения есть локально, но `deploy.sh` пишет "Изменений не обнаружено"

**Причина**: локальный `.deploy-manifest` содержит устаревшие `mtime/size`, либо браузер показывает закешированную версию статики.

**Решение**:
1. Очистите локальный манифест:
   ```bash
   rm -f .deploy-manifest
   ```
2. Повторите деплой:
   ```bash
   ./deploy.sh
   ```
3. Если нужно полное сравнение с бакетом:
   ```bash
   ./deploy.sh --full
   ```
4. После публикации сделайте hard refresh в браузере.

### Скрипт предупреждает, что Pillow не установлен

**Причина**: `start_server.py` и части deploy-пайплайна умеют работать без Pillow, но конвертация изображений в WebP в этом случае недоступна.

**Решение**:

```bash
python3 -m pip install Pillow
```

После этого повторите запуск сервера или деплоя.

### Локально интерфейс открывается, но часть действий не работает через `python3 -m http.server`

**Причина**: простой HTTP-сервер не реализует дополнительные обработчики из `start_server.py`.

**Решение**: для полноценной локальной работы запускайте:

```bash
python3 start_server.py
```

## Проверка настройки

После настройки проверьте, что всё работает:

```bash
# Проверка доступа к бакету
aws --endpoint-url=https://storage.yandexcloud.net s3 ls s3://practicum-banners

# Проверка загрузки файла
aws --endpoint-url=https://storage.yandexcloud.net s3 cp test.txt s3://practicum-banners/test.txt

# Проверка публичного доступа (должен открыться без авторизации)
curl https://practicum-banners.storage.yandexcloud.net/index.html
```

## Получение помощи

Если проблема не решена:
1. Проверьте логи ошибок в терминале
2. Убедитесь, что используете последнюю версию AWS CLI
3. Проверьте документацию Yandex Cloud: https://cloud.yandex.ru/docs/storage/
